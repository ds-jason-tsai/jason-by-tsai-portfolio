GA4 匯出至 BigQuery 的資料結構
當資料從 GA4 被匯出至 BigQuery 後，便要開始思考如何進行數據清洗，以利後續針對用戶的數位軌跡進行分析。本文說明 GA4 資料表各欄位的定義，並將其分為「事件」和「使用者」兩大主軸。
首先我們知道 GA4 是以「天」為單位，將資料批次匯入 BigQuery(以下簡稱 BQ) 當中。而經過筆者實測以及總結各界經驗後發現，GA4 的數據彙整需要有 2 天左右的緩衝。這邊的緩衝是基於部份使用者的數位軌跡在匯入 GA4 時會有延遲的情形，因此建議在分析時，篩選最新至 D-2 的日期較為恰當。

BQ 的結構定義由上圖可以發現，GA4 資料中的某些欄位是以 RECORD 的型態儲存(這部份後續會詳加說明)、且欄位數量頗多。而實際上，GA4 匯出至 BQ 的資料大致可分為事件和使用者兩類。關於 GA4 事件的說明，筆者之前有稍做整理，有興趣的朋友可以點擊這邊。
那事不宜遲，我們先來看看事件的部份有哪些資訊：
事件 Event
event_date | STRING | 事件時間(格式為 YYYYMMDD )
event_timestamp | INTEGER | 事件時間(以 UTC 紀錄至微秒)
event_previous_timestamp | INTEGER | 事件前一次被記錄的時間
event_name | STRING | 事件名稱
event_value_in_usd | FLOAT | 將事件中 value 參數轉為美元的數值
event_bundle_sequence_id | INTEGER | 事件被上傳時的序列 ID
event_server_timestamp_offset | INTEGER | 事件從蒐集到上傳的時間差

GA4 資料在匯入 BigQuery 後會自動以 event_date 來區隔資料表。透過上方敘述可知，event_bundle_sequence_id 和 event_server_timestamp_offset 兩個欄位與後續分析的關聯性較低，比較實用的可能是 2–5。
除上述欄位之外，事件的參數會以 BQ 當中的 STRUCT 格式被記錄。BQ 透過這種巢狀的資料型態將同一事件的多組 {key(param_name): value} 儲存在同一筆紀錄當中，具體如下：
event_params 的巢狀結構event_params.key | STRING
event_params.value | RECORD | 存放 event_params.key 對應的參數值
event_params.value.string_value | ex: URL
event_params.value.int_value | ex: ga_session_id
event_params.value.double_value
event_params.value.float_value

實際上，一個事件參數(event_params.key)的 value 只會對應到一種特定的資料型態。如 event_params.key = page_location 時，只有 string_value 會顯示其對應的 URL，而其它諸如 int_value、double_value 以及 float_value 則顯示為 null。
「事件」的流量來源 Collected_Traffic_Source
manual_campaign_id | STRING | 廣告活動 ID (UTM_ID)
manual_campaign_name | STRING | 廣告活動名稱(UTM_Campaign)
manual_source | STRING | 廣告活動來源(UTM_Source)
manual_medium | STRING | 廣告活動媒介(UTM_Medium)
manual_term | STRING | 廣告活動關鍵字(UTM_Term)
manual_content | STRING | 廣告活動中繼資料(UTM_Content)
gclid | STRING | Google Click ID
dclid | STRING | Google Marketing Platform (GMP) ID
srsltid | STRING | Google Merchant Center ID

在上述 9 個欄位當中，編號 1–6 的變數是較長被使用的。其中又以 Source & Medium 最為重要。假設我們在 Line OA 透過輪播版位的方式來投放廣告，Source 可以寫作 Line、而 Medium 則是 Carousel。關於 UTM 設計，筆者在這邊不多做贅述，但良好的參數管理在實務上確實能帶來諸多好處。

接著來看看使用者的部份有哪些資訊：
使用者 User
user_id | STRING | unique ID (需透過 GTM 蒐集，預設為 null )
user_pseudo_id | STRING | 匿名 ID
is_active_user | BOOLEAN | 以天為單位，紀錄使用者的活躍與否
user_first_touch_timestamp | INTEGER | 使用者初次造訪網頁或瀏覽 APP 的時間點

值得一提的是，user_pseudo_id 是由兩組數字所構成(中間以 . 隔開)，如：1806154613.1698723554。其中，前方的 1806154613 是隨機亂碼、後方的 1698723554 則是使用者初次造訪網頁時的 ga_session_id。
另外，在什麼情況下 is_active_user 會被判定為 True 呢？根據 Google 官方文件說法：當使用者為新使用者或在當日有互動工作階段(Engaged Session)時，is_active_user 就會保持 True 的狀態。
而工作階段要成為互動工作階段則需滿足下列條件的任意一項：
持續 10 秒以上
瀏覽 2 個以上的網頁(或 APP 畫面)
觸發 1 次以上的轉換事件

最後，GA4 提供的 user_first_touch_timestamp 這個變數可說是非常方便，未來分析若要計算各使用者初次造訪的距今天數(如 RFM 當中的 Recency )就能夠更快地完成。
裝置 Device
在 BigQuery 當中，Device 透過 STRUCT 的型態儲存，子項目如下：
device.category | STRING | 裝置類型(手機、平板、電腦、Smart TV 等)
device.mobile_brand_name | STRING | 裝置品牌(Apple、Google 等)
device.mobile_model_name | STRING | 型號名稱(iPad Air 2、Pixel 7 等)
device.mobile_marketing_name | STRING | 行銷名稱(如 Asus Zenfone)
device.mobile_os_hardware_model | STRING | 作業系統的裝置型號資訊
device.operating_system | STRING | 作業系統(iOS、Windows 等)
device.operating_system_version | STRING | 系統版本(如 iOS 16.6.1)
device.vendor_id | STRING | 供應商識別碼(IDFV, Identifier For Vendor)
device.advertising_id | STRING | 廣告識別碼
device.language | STRING | 作業系統語言(en-us、zh-tw 等)
device.time_zone_offset_seconds | INTEGER | 與 GMT 的時差(秒)
device.is_limited_ad_tracking | BOOLEAN | 裝置是否限制廣告追蹤
device.web_info.browser | STRING | 瀏覽器(Chrome、Safari 等)
device.web_info.browser_version | STRING | 瀏覽器的版本
device.web_info.hostname | STRING | 與事件相關聯的主機名稱

關於使用者裝置的欄位看似很多，但在實務上僅需針對特定的分析任務撈取對應的資訊即可。舉例而言，如果主管想了解不同數位平台使用者的裝置和品牌差異，使用 category 加上 mobile_brand_name 或許就能夠滿足需求。另外在異常流量的監控上，透過 GA4 的 Device 系列變數也能為我們提供很大的幫助。
地理位置 Geo
GA4 以事件發生當下的 IP 位址為依據儲存使用者地理位置的相關資訊：
geo.continent | STRING | 洲別(即五大洲)
geo.sub_continent | STRING | 子洲別(東亞、南亞、東南亞等)
geo.country | STRING | 國家/地區(日本、韓國、台灣等)
geo.region | STRING | 區域(台北市、台中市等)
geo.metro | STRING | 都會區(如華盛頓州的 Tri-Cities)
geo.city | STRING | 城市(竹北、中壢等)

由此可知除了 geo.metro 外，1–6 的地理資訊之間是有階層關係的。另外，由於 VPN 的盛行和種種因素，地理位置的準確度或有可議之處。這邊大家可以一同測試看看。
最後看到網站(App)初次招攬到「使用者」流量來源的部份，GA4 很貼心的紀錄了我們初次獲客時的流量資訊，這點與先前使用者(User)相關欄位中的 event_previous_timestamp 有異曲同工之妙，可見 GA4 對獲客資訊的重視程度。
初次招攬到「使用者」的流量來源 Collected_Traffic_Source
traffic_source.name | STRING | 行銷廣告的活動名稱
traffic_source.medium | STRING | 媒介名稱(ex: paid, organic search)
traffic_source.source | STRING | 聯播網名稱

除前述提到的事件(Event)以及使用者(User)兩大區隔外，我將 stream_id 及 platform 編入「其他」。實際上大家的網頁若販售商品，GA4 還提供如商品名稱(item_name)、商品價格(price)等電商相關的欄位，筆者的網站則因為沒有相關的資訊而自行省略。
其他 Others
stream_id | STRING | 事件來源的資料串流 ID
platform | STRING | 事件來源的資料串流平台(Web、iOS、Android 等)

透過這篇文章，我們已經簡單了解到 GA4 資料表當中 40+ 欄位的定義。而 GA4 也透過 event、user、device、geo、traffic 等維度區隔為數眾多的數位軌跡資料。
以上是 GA4 將 raw data 匯入 BQ 後資料樣態的一點分享以及欄位說明。如果想看更多資訊，歡迎到我的個人網站逛逛。
